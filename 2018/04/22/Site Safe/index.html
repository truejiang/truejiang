安全协议的防御方式

Posted on 2018-04-22 | In 网站安全 | Visitors:

一，xss 跨站脚本攻击

分类：
1，反射型（url 参数直接注入）
如： localhost:8080/<script>alert(1)</script>
2，存储型（存储到DB后读取时注入）
把恶意脚本存储到攻击目标的数据库中，等待猎物点击
如：发表一个评论时写的内容是: 哈哈<script>alert(1)</script>
用户在读取时就会触发脚本内的 alert(1)

注入点：
1，HTML节点内容
如：JSP或者一些模版语言，或者引入页面时使用了 innerHTML，又或者Vue的v-html等
<div>#{content}</div> contont是 一段脚本时。
2，HTML属性（提前终止属性）
如：上传了这样的一段代码，<img src="1" onerror="alert(1)"> 和上面一样在img读取不出来的时候就会触发后面onerror里面的脚本
3，富文本
因为富文本带有HTML，所以和上面2个的一样，HTML带有XSS攻击的风险

XSS防御方法：
1，打开浏览器自带防御，只针对反射性XSS
在http协议的header中写 header(‘X-XSS-Protection’, 1)

2，HTML内容的防御
用户发表上传的内容，在进入数据前转义
< 转义 &lt; > 转成 &gt; 等
如： <script><script> 转移成 &lt;script&gt;&lt;/script&gt;

3，HTML属性的防御
空格，单引号，双引号转义 转义 “ &quto;

4，JavaScript的防御
转移 “\” 或者转换成json

5，富文本防御
因为js脚本可以藏在很多地方，比如标签，超链接url里或者其他任何属性里
如:<script>alert(1)</scirpt><a href="javascript:alert(1)"></a>
解决办法可以过滤掉间的内容，还可以限制只允许一些标签和属性的存入

XSS防御总结：
前端尽量不用HTML插入的方式，后端开起浏览器自带防御，存入DB时内置函数转义，设置DOM解析黑白名单，辅助使用一些XSS防御的第三方库等等。

二，CSRF 跨站请求伪造攻击

含义：
csrf是让用户在不知情的情况，冒用其身份发起了一个请求
点击不知情链接，图片url.src

原理：
1，用户登录了A网站
2，A网站确认了用户身份
3，B网站页面向A网站请求（带了A网站的身份）

防御方法：
1，减少全站的cookie使用，设置cookie域，禁止其他网站带cookie的请求
2，通过 referer（禁止第三方网站的请求），token，验证码等方式，验证用户的表单提交
3，对于用户的增删改等操作都使用POST请求
4，尽量不要在页面的链接中暴露用户信息
5，same-site属性（兼容性不高，只有chrome和oppo支持）
参考链接：
https://www.ibm.com/developerworks/cn/web/1102_niugang_csrf/

后台防御方法：
1，Cookie samesite属性
header(‘Set-Cookie: test: 12345; SameSite=Lax’)
2，HTTP referer头
3，设置token

三，DOS防御

防御方法：
1，防火墙
2，交换机，路由器
3，流量清洗
4，高防IP
5，避免重复编辑业务
6，快速失败快速返回
7，防雪崩机制
8，有损服务
9，CDN

四，Cookie

XSS可以偷取Cookie
防御：
1，通过设置 http-only 的 Cookie 不会被偷取

当您使用HttpOnly标记标记Cookie时，它会告知浏览器，该特定Cookie只能由服务器访问。 任何尝试从客户端脚本访问cookie是严格禁止的。 当然，这假设你有：一个现代的网络浏览器。 一个实际上正确实现HttpOnly的浏览器。
2，设置 same-stie 组织第三方使用Cookies

五，点击劫持和传输安全

点击劫持防御方法：
1，Javascript 禁止内嵌
通过 top === Window 是否为true对比。带frame的页面为false
2，H5属性禁止运行内嵌
sandbox=”allow-forms”
3，后台设置禁止内容
header(‘X-FRAME-OPTIONS’, ‘DENY’)
4，加验证码

HTTP验证的，看的不是很明白，参考下面的页面
https://blog.csdn.net/qq_33936481/article/details/77604344

六，密码管理

解决方法：
1，密码加密（哈希算法）
有md5 sha1 sha256 等

七，接入层（SQL）注入
防御：
1，关闭错误输出（判断注入是否正确）
2，检查数据类型,参数（parseInt()）
3，对数据进行转义（npm install mysql 手工麻烦）
4，使用参数化查询（npm install mysql2）
5，使用ORM（对象关系映射）
参考文章：
https://blog.csdn.net/qq_33936481/article/details/77618896

八，上传问题防御

1，限制上传后缀
欺骗，改后缀
2，文件类型检查
容易欺骗（浏览器检查，可以使用其他工具上传）
3，文件内容检查（文件头信息）
欺骗（先把头信息写入，后面写程序）
4，程序输出(读，写，性能有影响)
